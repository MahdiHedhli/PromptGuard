# DEC-026: Real-corpus benchmark validation (AI4Privacy + GitHub Issues Secrets)

**Date:** 2026-05-02
**Status:** Accepted
**Phase:** v1.1.1 (Phase 7)
**Author:** MahdiHedhli

---

## Context

The v1.1 release shipped detection numbers from a 220-example synthetic
corpus generated by `benchmarks/run_detection_benchmarks.py`. Owner
review surfaced that synthetic-only numbers land soft for the blog
audience and that the public-release decision should gate on
real-corpus validation. v1.1.1 closes that gap.

## What was done

### Corpora

Two real public corpora, accessed under their respective licenses,
not redistributed in the repo, SHA256-manifested for reproducibility:

* **AI4Privacy PII-Masking-300k** (Hugging Face: `ai4privacy/pii-masking-300k`).
  Validation split, English subset only (7,946 records of 47,728 total
  validation records). Manifest at
  `local/benchmarks/corpora/ai4privacy/manifest.txt`. Used under the
  AI4Privacy "other" license: academic evaluation use without separate
  licensing, commercial training requires licensing. We use only for
  evaluation; we do not train.
* **GitHub Issues Secrets Benchmark** (Zenodo: `https://zenodo.org/records/17430336`,
  arxiv 2410.23657). `test_wild.csv` split, 1,488 records, 29 labeled-real
  secrets and 1,459 hard-negative candidates. Released CC-BY 4.0.
  Manifest at `local/benchmarks/corpora/github-secrets/manifest.txt`.
  Used under the CC-BY license; cited per the requirement.

### Harness

`benchmarks/run_real_corpus_benchmarks.py`. Runs three pipelines
against each corpus and computes per-category precision / recall / F1
using span-IoU >= 0.5 as the match criterion.

Pipelines:

1. **`baseline_litellm`**: stand-in for LiteLLM's native PII pattern
   matching. Minimal regex set (email, IPv4, AWS access key shape,
   generic 32+ hex SECRET) implemented inline so the comparison is
   reproducible from this file alone.
2. **`promptguard_regex`**: PromptGuard's regex layer alone with
   v1.1 NormalizationDetector.
3. **`promptguard_full`**: regex + normalization + OPF + Presidio.
   Detector availability probed at runtime; output records which
   detectors were actually present.

### Mapping discipline

Three category-mapping decisions documented for honesty:

* **AI4Privacy `IP` -> PromptGuard `INTERNAL_IP`, RFC-1918 only.** The
  corpus's `IP` label covers public IPs and RFC-1918 internal IPs
  uniformly. PromptGuard's `INTERNAL_IP` detector targets RFC-1918 by
  design (public IPs are not the threat surface; they appear in URLs,
  log lines, public documentation). Public-IP gold spans are filtered
  out of the eval rather than counted as misses, which would overstate
  the gap.
* **AI4Privacy `USERNAME / IDCARD / SOCIALNUMBER / PASSPORT /
  DRIVERLICENSE / BOD / DATE / TIME / SEX / TITLE / PASS / GEOCOORD /
  CARDISSUER`** are explicit out-of-scope. PromptGuard v1.1.1 does not
  attempt these categories. They are excluded from scoring rather than
  reported as F1=0 against fair-recall denominators that do not exist.
* **GitHub Issues Secrets vendor-specific types -> `SECRET`.** The
  corpus's `secret_type` field carries fine-grained vendor labels
  (Lob, Anypoint, Sparkpost, Auth0). PromptGuard does not split SECRET
  into per-vendor subcategories. We score under SECRET as the most
  generous mapping; per-vendor splits would not change the headline.

## Findings worth surfacing

### What landed cleanly

* **PromptGuard's RFC-1918 IP detection: F1 1.000 with zero FPs.** The
  baseline catches RFC-1918 IPs at the same recall but floods the
  surface with 1,115 false positives on public IPs. The "deterministic
  detection on the categories CISOs lose sleep over" claim survives
  contact with the real corpus.
* **Email detection: F1 0.96** for both PromptGuard regex-only and the
  baseline. Email is a genuinely shape-detectable category and the
  vendored regexes work.
* **Free-form private-name / private-address / private-phone via
  full pipeline: F1 0.27-0.36, recall 0.23-0.40.** Numbers are
  consistent with Tonic.ai's published OPF default-operating-point
  results on out-of-distribution data (R around 0.10-0.38). The full
  pipeline picks up categories the regex layer cannot see by shape.

### Two non-obvious measurement artifacts

* **Full-pipeline email apparent-F1 dip (P 0.49).** OPF and Presidio
  also flag emails with slightly different span boundaries; the
  harness scores raw detections without action-engine deduplication.
  The action engine's `_select_outer_spans` (DEC-023) collapses these
  to one substitution at runtime. This is a measurement artifact, not
  a regression; operator-visible behavior is correct. v1.2 candidate:
  add a score-time dedup pass that mirrors the action engine.
* **Full-pipeline IP F1 0.004.** Same root cause: OPF and Presidio
  flag the 4 RFC-1918 IPs with category=`OTHER` rather than
  `INTERNAL_IP`. The harness scores categories strictly. Action engine
  dedup at runtime gives the operator a correct INTERNAL_IP
  substitution. v1.2 candidate: a category-fold pass for
  cross-detector category disagreements.

### The honest gap

* **Vendor-specific secrets: F1 0 across all pipelines.** The 29 real
  secrets in `test_wild` are predominantly Anypoint / Lob / Sparkpost
  / Auth0 / Artifactory keys that have no distinctive prefix. Our v1.1
  regex set targets the credential categories the gitleaks ruleset
  converged on; it does not cover the long tail. The baseline catches
  one of them but generates 1,553 FPs on every commit SHA / SHA-256
  hash / long hex string in the corpus, which is unusable for an
  inline blocking proxy.

  **This is calibration, not architecture.** Two v1.2 fixes:

  1. Extend the regex set with vendor-specific patterns from a broader
     gitleaks rule audit (gitleaks v8.x ships rules for many of these).
  2. Ship Presidio custom recognizers configured per-deploying
     organization. Anypoint key detection is identifying for an org
     that uses Anypoint; an org that does not has no use for the rule.
     This is structurally what the Presidio adapter is for; v1.1 does
     not configure these out of the box.

  Per the v1.1.1 brief: "do NOT tune to close the gap; document
  honestly and flag for owner review." That is the call here. The
  gap is real, the framing is honest, the fix path is identified.

## Decision

Real-corpus benchmarks become the credibility claim. Synthetic-corpus
results stay in the codebase as development-time regression targets
and are explicitly framed as such in `docs/benchmarks.md`. Public
documents (README, blog draft, NOTICE) reference the AI4Privacy +
GitHub Issues Secrets results.

The OPF recall-tuned operating point sweep, the cross-detector
category-fold pass, and the vendor-specific recognizer pack are
v1.2 work items. v1.1.1 ships with the honest numbers as-is.

## Consequences

### Enables
* The blog-quality claim ("PromptGuard caught every RFC-1918 IP in
  AI4Privacy English at F1 1.000 while a v0 baseline floods 1,115
  false positives") rests on a real, reproducible benchmark.
* Operators evaluating the project against published numbers can
  re-run the harness and verify the result with one command.
* The methodology is reproducible: corpora are pinned by SHA256, the
  harness is in the repo, the mapping decisions are explicit.

### Constrains
* The vendor-specific secrets gap is publicly visible. Operators who
  evaluate against the GitHub Issues Secrets benchmark see F1 0 on a
  category they may care about.
* The full-pipeline measurement artifacts (email FP inflation, IP
  category fold) require a docs explanation; they look like
  regressions on a casual reading.
* Synthetic numbers are no longer the credibility claim. Future
  development-time corpus changes do not affect blog credibility,
  but real-corpus regressions do.

### Revisit if
* AI4Privacy or the GitHub Issues Secrets benchmark changes license
  terms. Both are documented as eval-only; commercial training is
  out of scope for both.
* A vendor-specific recognizer pack lands and shifts the GitHub
  Issues Secrets numbers materially.
* The OPF recall-tuned operating point sweep (v1.2) shifts the
  full-pipeline numbers on AI4Privacy private-name / address / phone.
