=== Real Anthropic round-trip via PromptGuard proxy ===
=== Date: 2026-05-01 ===
=== Policy: regex-only ===
=== Stack: PromptGuard proxy + real api.anthropic.com upstream ===

This is the headline TOKENIZE-round-trip artifact, captured against
api.anthropic.com (claude-sonnet-4-6).

User prompt sent to the proxy (HTTP request body):
  {"model":"claude-sonnet-4-6","max_tokens":80,
   "messages":[{"role":"user","content":
     "Reply with exactly: server 10.0.13.42 ok"}]}

What the upstream Claude actually saw (PromptGuard tokenized 10.0.13.42):
  user content: "Reply with exactly: server [INTERNAL_IP_<16-hex>] ok"

User-visible response (PromptGuard restored on the way back):
  server 10.0.13.42 ok

Cost on the wire (Anthropic billing):
  input_tokens:  28
  output_tokens: 20

The IP never reached api.anthropic.com. The user got their original IP
back in the reply because PromptGuard's reverse path substituted the
random per-conversation token for the original value before the bytes
left the proxy.

== Compatibility note (Day 8 finding) ==

The same round-trip via the `claude` CLI (`claude --print "..."`) does
NOT work on prompts that trigger our outbound rewriting. The CLI's
streaming-response parser rejects rebuilt SSE event sequences with
"Content block is not a text block". Direct HTTP requests (curl,
SDKs, apps that respect ANTHROPIC_BASE_URL with their own parser)
work cleanly. Tracked as a v1.1 compatibility item; root cause is
in claude CLI v2.x's stricter event-shape expectations on the
streaming response, not in PromptGuard's request-side rewriting.
