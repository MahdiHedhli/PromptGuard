WHAT THE UPSTREAM CLAUDE ACTUALLY SAW:
  PromptGuard's pre-call hook tokenized 10.0.13.42 before forwarding.
  user content: "...the IP [INTERNAL_IP_<16-hex>] fall into?..."

USER-VISIBLE TERMINAL OUTPUT:
10.0.0.1 falls into the `10.0.0.0/8` private network range (RFC 1918 Class A).

EVIDENCE OF SUCCESSFUL TOKENIZATION:
  Note that Claude's reply references `10.0.0.1`, NOT the user's
  `10.0.13.42`. Claude could not see the original IP. It saw a token,
  decided the prompt was asking about a generic RFC-1918 example, and
  produced a different example IP from the same range. This is
  positive evidence that PromptGuard's tokenization worked: a model
  that had access to `10.0.13.42` would have echoed that exact value.

ROUND-TRIP CAVEAT:
  Token round-trip restoration works WHEN the upstream LLM echoes the
  token verbatim in its reply. When the LLM paraphrases (as Claude
  did here, replacing the abstract token with a concrete example),
  there is nothing for the reverse path to substitute and the user
  sees the LLM's generated text. Operators who depend on round-trip
  fidelity for a category should test their typical prompts against
  their target model; high-temperature or summarization-style prompts
  paraphrase more aggressively than literal-quote prompts. The
  deterministic stages (BLOCK and MASK) are unaffected by this; only
  TOKENIZE depends on the model's paraphrase behavior.
