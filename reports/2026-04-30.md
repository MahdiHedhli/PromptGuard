## Code / artifact highlights (blog post fodder)

- **Threat-model entry A6 (token-map ledger as attack surface).** This is the framing that genuinely separates PromptGuard from Rama's tutorial. MASK is recommended over TOKENIZE for any data class where the ledger itself is contractual or regulated; the explicit calling-out of "the round-trip mechanism is itself a sensitive datastore" is a real CISO-grade observation. See [docs/threat-model.md A6](../docs/threat-model.md).
- **Multi-stage detector layering.** The architecture diagram in [docs/architecture.md](../docs/architecture.md) shows regex / OPF / Presidio / LLM-judge as composable stages. The blog can lead with: "regex catches shapes, OPF catches context, Presidio catches your codenames, LLM judge catches paraphrasing." The Tonic.ai benchmark (research-notes section 6: OPF default-recall on out-of-distribution data is 10% on web crawls, 38% on EHR) is the single most compelling argument for layering.
- **Per-pattern attribution.** Every `PatternSpec` in [regex_patterns.py](../src/promptguard/detectors/regex_patterns.py) has a `source` field. This is overkill for a regex list, but it is exactly the kind of thing a security-conscious consumer expects.
- **Health vs readiness split for OPF.** `/health` reports server-up immediately; `/ready` reports model-loaded only after the first inference path warms. This is operationally correct (don't block startup on a 3GB pull) and is explained in [DEC-002](../decisions/002-opf-service-shape.md). Worth a one-paragraph explainer in the post.
- **Quick number for the post:** RegexDetector microbenchmark, all 17 patterns over a 4,640-char prompt: **~1.05 ms per call** on Apple Silicon (M-series) Python 3.11. The regex layer is effectively free at any reasonable prompt size.
