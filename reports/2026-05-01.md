## Code / artifact highlights (blog post fodder)

- **The OPF hard-fail message itself is a teachable moment.** "PromptGuard's promise is that PII never leaves the machine; silently falling back to a model with weaker coverage breaks that promise without the operator knowing" is the line that makes a CISO nod. The actual error message tells the operator how to opt out (`detectors.opf.enabled = false`), so the right thing happens by default and the escape hatch is documented at the failure site, not in a wiki page. See [DEC-009](../decisions/009-opf-eager-load-and-hard-fail.md).
- **The BLOCK envelope's "no payload echo" guard.** [errors.py](../src/promptguard/proxy/errors.py) ships a defensive `assert_no_payload_leak` helper that walks 6-character windows of the original prompt against the serialized envelope and raises if any window appears. It's the kind of belt-and-braces invariant you can't talk anyone out of in a security review. The unit test "test_assert_no_payload_leak_detects_leakage" sets up a deliberately leaky envelope and checks the assertion fires; that's the test the post can show.
- **JSON-safe boundary trick.** Concatenate inspected strings with `\x00\x00PG_BOUNDARY\x00\x00`, run detection once, split back. One pipeline call per request, envelope keys untouched, action engine's overlap handling stays consistent. Worth a paragraph in the post: "the obvious approach is detect-per-string; the right approach is detect-once-per-request because that's how the overlap handling stays right." See [DEC-011](../decisions/011-json-safe-substitution.md).
- **Latency numbers worth citing:**
  - In-process Day 1 baseline (RegexDetector only): avg 0.071ms / p95 0.075ms.
  - In-process Day 2 path (pipeline + action engine + JSON-safe extract/rewrite): avg 0.154ms / p95 0.271ms / p99 0.487ms.
  - Net overhead from Day 1: +0.083ms avg. Action engine + JSON-safe rewrite is essentially free on a 305-char prompt.
  - End-to-end through proxy on BLOCK path (LiteLLM in front of the hook): avg 12.4ms / p95 17.6ms / p99 17.6ms. The bulk of that is LiteLLM's request handling, not PromptGuard.
- **The policy-error UX.** Instead of "validation error", operators get `at line 7, column 12 (rules.2.action): Input should be 'BLOCK', 'MASK', 'TOKENIZE' or 'ALLOW'; got 'REJECT'`. They can navigate to the broken line. The line-tracking SafeLoader in [local_yaml.py](../src/promptguard/policies/local_yaml.py) is small enough to lift into the post if anyone asks "how is this implemented".
