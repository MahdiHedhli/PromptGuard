# PromptGuard MITM verification harness

Independent wire verification: see what bytes actually leave your machine when you run prompts through PromptGuard.

This harness positions a `mitmproxy` container **between LiteLLM and the upstream provider** (api.anthropic.com or whichever upstream LiteLLM is configured for). It is NOT positioned between the client and LiteLLM. The point of placement is intentional: you already know what the client sent in (you wrote the prompt). What you want to verify is what PromptGuard let out.

The harness is **opt-in**. The default `docker compose up` flow does not include it; you have to invoke the overlay explicitly.

## TL;DR

```bash
# Bring up proxy + mitmproxy overlay, all in one command.
make -C tools/mitm-verify up

# Run the canned prompt suite and produce a summary report.
make -C tools/mitm-verify test

# Inspect captures or stop the stack.
make -C tools/mitm-verify logs
make -C tools/mitm-verify down

# Scrub captures (does not touch git-tracked files).
make -C tools/mitm-verify scrub
```

The summary report lands at `./local/mitm-captures/summary-<UTC-timestamp>.md`. Per-flow capture files (one per request, one per response) land in the same directory.

`./local/` is gitignored. Captures never leave your machine unless you copy them out.

## What the harness does

1. **mitmproxy as upstream proxy.** A container running `mitmdump` on port 8080 inside the `promptguard` docker network. LiteLLM gets `HTTPS_PROXY=http://mitmproxy:8080` so all outbound HTTPS traffic from LiteLLM goes through mitmproxy first.

2. **Auto-generated CA cert at `./local/mitm-ca/`.** First run creates the CA. The LiteLLM container mounts this CA as `SSL_CERT_FILE` and `REQUESTS_CA_BUNDLE`, so it trusts the proxy's TLS interception. Re-running the harness regenerates the CA only if you scrubbed it; otherwise the existing CA is reused.

3. **Per-flow capture files.** [`addon.py`](addon.py) is loaded into mitmdump on startup. For every request and every response, it writes a JSON file at `./local/mitm-captures/<UTC>-<flow-id>-{req,resp}.json` with:
   - full URL and method
   - HTTP headers (Authorization and x-api-key redacted to last 4 chars)
   - body size in bytes
   - prettified body (parsed JSON if possible, raw text otherwise)
   - a `suspicious_pattern_counts` block listing any obvious-PII shapes the body contained (raw RFC-1918 IPv4, raw email, raw PEM marker, raw AWS access key)

4. **Console output.** Each request emits one line: timestamp, method + URL, body size, and a flag if any suspicious pattern matched. The flag is a SANITY HEURISTIC, not detection. The point is "did anything obvious leak."

5. **Run-suite script.** [`run-test-suite.sh`](run-test-suite.sh) runs every prompt in `test-prompts/` through the proxy at `localhost:${PROMPTGUARD_LITELLM_PORT:-4100}` and produces a single Markdown summary at `./local/mitm-captures/summary-<UTC>.md`. The summary is the single artifact you read after a verification run.

## Why this is the right tool

The threat-model promise is "PII never leaves the host." Our test suite asserts that promise; benchmarks measure the detection F1; the audit log carries forensic events. None of those things show you what actually went out on the wire.

This harness shows you. The mitmproxy capture is positioned downstream of every PromptGuard hook. If a regression introduces a path that bypasses the rewrite or restoration logic, the capture catches it. If the threat model says "the offending text never reaches the upstream," the capture proves or disproves that on a per-prompt basis.

## Prerequisites

- Docker Desktop or Docker Engine, with the `promptguard` stack already buildable (i.e. `docker compose build` works).
- `.env` populated as documented in the repo root README (`ANTHROPIC_API_KEY` for real upstream, optional but recommended; otherwise the upstream returns 401 and you only see PromptGuard-side BLOCK envelopes).
- `make`, `bash`, `jq` (for inspecting captures with one-liners; not required for the harness itself).

## What gets persisted, what is gitignored

| Path | What is there | Gitignored? |
|---|---|---|
| `./local/mitm-ca/` | Auto-generated mitmproxy CA + key | yes (via `local/`) |
| `./local/mitm-captures/<ts>-<id>-req.json` | Per-flow request captures | yes |
| `./local/mitm-captures/<ts>-<id>-resp.json` | Per-flow response captures | yes |
| `./local/mitm-captures/summary-<ts>.md` | Per-run summary report | yes |
| `tools/mitm-verify/addon.py` | mitmproxy addon source | NO (tracked) |
| `tools/mitm-verify/docker-compose.mitm.yml` | Compose overlay | NO (tracked) |
| `tools/mitm-verify/test-prompts/*.txt` | Canned test prompts (synthetic only) | NO (tracked) |

Sample prompts use only synthetic data. The PEM block in `05-private-key.txt` is freshly generated by openssl on the day the file was created and does not correspond to any real certificate or service. AWS keys use the documented `AKIAIOSFODNN7EXAMPLE` example. Email domains use RFC-2606-reserved forms.

## Inspecting captures

Each capture file is one JSON object. The fields are documented at the top of [`addon.py`](addon.py).

A few useful jq one-liners:

```bash
# Show every request body that contained a literal IPv4.
jq 'select(.suspicious_pattern_counts.raw_ipv4_rfc1918 // 0 > 0)' \
   ./local/mitm-captures/*-req.json

# Show every response from api.anthropic.com.
jq 'select(.request_url | contains("anthropic.com"))' \
   ./local/mitm-captures/*-resp.json

# Sum up suspicious-pattern counts across an entire run.
jq -s '[.[] | .suspicious_pattern_counts | to_entries[]] |
       group_by(.key) | map({(.[0].key): (map(.value) | add)}) | add' \
   ./local/mitm-captures/*.json
```

## Failure modes

The harness is built to fail loudly rather than silently:

- If mitmproxy cannot write to `/captures` (missing volume mount, permission error), the addon raises and mitmdump refuses to start.
- If the CA file does not appear at `./local/mitm-ca/mitmproxy-ca-cert.pem` within the start period, `make up` fails the healthcheck rather than starting LiteLLM.
- If LiteLLM is not configured to trust the CA, outbound HTTPS to api.anthropic.com fails. You will see TLS errors in the LiteLLM container's logs, NOT a silent passthrough.
- The run-suite script exits non-zero if any test prompt produces a leak.

If a prompt **bypasses** the proxy (e.g. claude CLI hits the upstream directly), the mitmproxy container sees nothing and the suite reports "no upstream capture; expected for BLOCK." This is correct for genuinely BLOCKed prompts. For non-BLOCKed prompts that show no upstream capture, that is a red flag: someone has misconfigured the trust path. Run `make logs` and check for TLS errors before trusting the result.

## CA cert handling

The CA is per-machine and per-run. `make scrub` deletes it; the next `make up` creates a fresh one. There is no scenario in which you should share the CA across machines or operators. The CA's only purpose is to let your local LiteLLM container trust your local mitmproxy container.

If you previously installed an old version of the CA into your host system trust store, remove it before running the harness on a fresh checkout. The harness does NOT install the CA into your host trust store. Its scope is "what does my LiteLLM container send out," not "what does my host browser see."

## Tear down semantics

- `make down` stops containers, leaves networks / volumes alone where used by the parent compose, and preserves all captures.
- `make scrub` wipes `./local/mitm-captures` and `./local/mitm-ca`. The CA, captures, summary reports all gone. Next run creates a fresh CA.
- `docker compose down -v` from the repo root with the overlay would also remove the docker-compose-managed volumes; not needed here because all the persistence is on bind mounts under `./local/`.

## Adding a new test prompt

Drop a file at `tools/mitm-verify/test-prompts/NN-name.txt`. The convention:

- `NN` is a sortable two-digit prefix.
- The first lines are comment lines starting with `#`. The runner skips these.
- Use only synthetic data: documented examples, RFC-2606-reserved domains, openssl-generated PEMs.
- Document the expected behavior in the comment header.

Re-run `make test` to pick up the new prompt.
